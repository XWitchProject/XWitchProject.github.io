Detailed Description

Settings which control certain behavior of the parser.

◆ AutofillNumericForStep

bool Relua.Parser.Settings.AutofillNumericForStep = true

Automatically fills in the Step field of AST.NumericFor with a NumberLiteral of value 1 if the statement did not specify the step expression.

◆ AutofillSequentialKeysInTableConstructor

bool Relua.Parser.Settings.AutofillSequentialKeysInTableConstructor = true

Automatically creates NumberLiterals for sequential elements in a table constructor (ones that do not have a key specified).

Note that if this option is false, AST.TableConstructor.Entry's Key field may be null. That field will never be null if this option is set to true.

◆ AutofillValuesInLocalDeclaration

bool Relua.Parser.Settings.AutofillValuesInLocalDeclaration = true

Automatically creates NilLiterals for all values of empty local assignments (in the style of local a).

Note that if this option is false, AST.Assignment's Values list will be empty for local declarations. If it is set to the default true, the Values list will always match the Targets list in size in that case with all entries being NilLiterals.

◆ EnableLuaJITLongs

bool Relua.Parser.Settings.EnableLuaJITLongs = true

If true, will parse LuaJIT long numbers (in the form 0LL) into the special AST node AST.LuaJITLongLiteral.

◆ MaintainSyntaxErrorCompatibility

bool Relua.Parser.Settings.MaintainSyntaxErrorCompatibility = false

There are certain syntax quirks such as accessing the fields of a string literal (e.g. "abc":match(...)) which Lua will throw a syntax error upon seeing, but the Relua parser will happily accept (and correctly write). If this option is enabled, all Lua behavior is imitated, including errors where they are not strictly necessary.
